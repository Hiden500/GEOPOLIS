# LLM World Simulation Rules

## Основной принцип

LLM не является игровым движком.

LLM не рассчитывает экономику, производство, население, логистику, бои или другие игровые механики.

LLM принимает решения и создаёт события на основе данных, предоставленных игровым движком.

Игровой движок является единственным источником истины.

---

## Приоритеты

При генерации решений LLM обязана учитывать:

1. Интересы государства.
2. Внутреннюю ситуацию государства.
3. Экономические возможности.
4. Военные возможности.
5. Дипломатическую ситуацию.
6. Исторический контекст текущего года.
7. Только после этого действия игрока.

LLM не должна концентрироваться исключительно на игроке.

---

## Симуляция стран

Каждая страна должна рассматриваться как самостоятельный актор.

Каждый ход LLM обязана предполагать, что все государства продолжают:

- строить объекты;
- развивать экономику;
- исследовать технологии;
- развивать армию;
- проводить дипломатическую деятельность;
- реагировать на соседей;
- реагировать на мировые события.

Даже если игрок ничего не делает, мир должен продолжать развиваться.

---

## Ограничения знаний

LLM никогда не должна хранить карту мира в контексте.

LLM никогда не должна оперировать предположениями о существовании регионов.

Все регионы, страны, провинции, моря, океаны и объекты должны использоваться только из данных, переданных игровым движком.

Если регион отсутствует в переданных данных, он считается неизвестным.

---

## Работа с регионами

LLM никогда не должна использовать названия регионов как уникальные идентификаторы.

Все действия должны использовать идентификаторы объектов.

Пример:

Плохо:

"Построить завод в Уральской области"

Хорошо:

{
  "action": "build_factory",
  "regionId": "USSR-URAL-12"
}

---

## Работа с картой

LLM не должна создавать полигоны.

LLM не должна изменять геометрию карты.

LLM может только предлагать изменения игрового состояния:

- смена владельца региона;
- создание объекта;
- удаление объекта;
- изменение характеристик объекта;
- изменение характеристик страны.

**Правила создания Map Features:**

**Промышленность:**
- Заводы: только в регионах с соответствующими ресурсами (уголь, железо, нефть)
- Шахты: только в регионах с месторождениями ресурсов
- Верфи: только в прибрежных регионах
- Электростанции: рядом с потребителями (города, заводы)
- Исследовательские центры: только в крупных городах

**Военные объекты:**
- Армии: на территории владельца или в союзных регионах
- Флоты: в прибрежных регионах владельца
- Авиабазы: на территории владельца
- Военно-морские базы: только в прибрежных регионах
- Все военные соединения обязаны иметь тег `battalion`

**Инфраструктура:**
- Железные дороги: между регионами с экономической активностью
- Порты: только в прибрежных регионах
- Каналы: только между соответствующими водными объектами

**Правила удаления Map Features:**
- Удалять объекты только при оккупации или разрушении
- Не удалять автоматически без причины
- При смене владельца объекта сохранять его тип

---

## Масштабирование

Система должна корректно работать с:

- 2000–5000 регионами;
- 100+ государствами;
- десятками тысяч объектов карты.

LLM не должна требовать загрузки полного мира в контекст.

LLM должна работать только с релевантным подмножеством данных.

---

## Военные действия

LLM не должна самостоятельно рассчитывать результаты боя.

LLM должна принимать решения:

- наступать;
- обороняться;
- перебрасывать силы;
- мобилизовываться;
- заключать мир.

Результаты рассчитываются игровым движком.

---

## Экономика

LLM не рассчитывает ВВП.

LLM не рассчитывает производство ресурсов.

LLM не рассчитывает логистику.

LLM только принимает решения:

- строить завод;
- строить шахту;
- строить порт;
- строить железную дорогу;
- инвестировать в отрасль.

Все результаты рассчитываются игровым движком.

---

## Историческая достоверность

LLM обязана учитывать исторические реалии текущего года.

1946 год не должен использовать технологии 1980-х годов.

1880 год не должен использовать технологии XX века.

События должны соответствовать доступным технологиям, политическим условиям и уровню развития эпохи.

---

## Альтернативная история

После отклонения от реальной истории LLM должна руководствоваться логикой мира, а не пытаться возвращать события к историческому результату.

Игрок может изменить ход истории.

Другие государства должны реалистично реагировать на последствия этих изменений.

---

## Генерация событий

События должны возникать не только вокруг игрока.

Большинство событий должны происходить независимо от игрока.

Мир обязан создавать собственую историю.

Примеры:

- экономические кризисы;
- перевороты;
- выборы;
- гражданские войны;
- научные открытия;
- локальные конфликты;
- дипломатические кризисы;
- колониальные восстания.

Игрок является частью мира, а не центром вселенной.

---

## Формат ответа LLM

LLM обязана возвращать ответ в строго структурированном формате.

**Обязательная структура JSON:**

```json
{
  "turn": "1946-01",
  "events": [
    {
      "id": "event-uuid",
      "type": "war|crisis|discovery|election|coup",
      "date": "1946-01",
      "description": "...",
      "affectedCountries": ["countryId"],
      "effects": {...}
    }
  ],
  "countryDecisions": {
    "countryId": {
      "actions": [
        {
          "type": "build_factory|research_tech|declare_war",
          "targetId": "regionId|techId",
          "parameters": {...}
        }
      ]
    }
  },
  "worldStateChanges": {
    "regions": {...},
    "countries": {...}
  }
}
```

**Обязательные поля:**
- `turn` - текущий игровой ход (год-месяц)
- `events` - массив мировых событий
- `countryDecisions` - решения стран по ID
- `worldStateChanges` - изменения состояния мира

**Запрещено:**
- Возвращать свободный текст без структуры
- Использовать неопределённые поля
- Пропускать обязательные секции
- Использовать комментарии в JSON

---

## Ограничения объёма ответа

LLM обязана ограничивать количество изменений за один игровой ход.

### Динамические лимиты

Лимиты зависят от интервала симуляции (скачка).

**Базовые значения (для месяца):**
- Событий: 5-8
- Решений на страну: 1-2
- Новых Map Features: 15-20
- Изменений владельца регионов: 3-5

**Коэффициенты для интервалов:**
- Месяц: ×1.0
- Квартал: ×2.0
- Полугодие: ×3.0
- Год: ×5.0

**Пример расчёта для года:**
- Событий: 5-8 × 5.0 = 25-40
- Решений на страну: 1-2 × 5.0 = 5-10
- Новых Map Features: 15-20 × 5.0 = 75-100
- Изменений владельца регионов: 3-5 × 5.0 = 15-25

### Исторические события

Исторические события (с тегом "historical") при соблюдении условий происходят обязательно:
- Не учитываются в лимите количества событий
- Примеры: начало Второй мировой войны (1939), падение Берлинской стены (1989), распад СССР (1991)

### Действия игрока

Действия игрока обрабатываются отдельно от лимита LLM:
- Лимит действий игрока за ход: 10 (настраиваемый параметр)
- Не учитываются в лимите количества событий LLM
- Последствия действий игрока учитываются в общем состоянии мира

### Причины ограничений

- Контекст LLM ограничен
- Слишком много изменений сложно валидировать
- Игрок не должен теряться в событиях
- Изменения должны быть предсказуемыми

### Исключения

- Глобальные войны могут требовать больше изменений
- Крупные кризисы могут увеличивать количество событий

---

## Приоритеты генерации событий

LLM обязана следовать приоритетам при выборе событий для генерации.

**Высокий приоритет (всегда включать при наличии условий):**
- Войны и мирные договоры
- Экономические кризисы
- Перевороты и смены режимов
- Крупные научные открытия
- Ядерные испытания (после 1945 года)

**Средний приоритет (включать при возможности):**
- Дипломатические кризисы
- Выборы
- Локальные конфликты
- Колониальные восстания
- Территориальные споры

**Низкий приоритет (включать по остаточному принципу):**
- Рутинные события
- Второстепенные новости
- Малозначительные экономические изменения

**Правило приоритета:**
При ограничении количества событий LLM должна предпочитать события высокого приоритета событиям низкого приоритета.

---

## Критерии валидности решений

Игровой движок должен отклонять решения, не соответствующие игровым правилам.

**Игровой движок должен отклонять:**
- Решения без достаточных ресурсов (manpower, treasury, materials)
- Использование недоступных технологий для текущего года
- Создание объектов в несуществующих регионах
- Невозможные дипломатические действия (союз с врагом, гарантия независимости без ресурсов)
- Переброска армии через океан без флота
- Строительство объектов без необходимых условий (завод без ресурсов, верфь без моря)

**LLM должна проверять перед принятием решения:**
- Достаточность ресурсов (treasury, manpower, materials)
- Соответствие технологий текущему году
- Существование целевых регионов в переданных данных
- Логическую обоснованность дипломатических действий
- Наличие необходимых условий для строительства

**При невозможности выполнить действие:**
- LLM должна предложить альтернативное действие
- LLM должна отложить действие на будущий ход
- LLM должна объяснить причину невозможности

---

## Минимальный контекст промта

Игровой движок обязан включать в промт минимальный набор данных для корректной работы LLM.

**Обязательные данные в каждом промте:**
- Текущая дата (год-месяц)
- Список стран с базовыми характеристиками (id, название, идеология, treasury, manpower)
- Список регионов с владельцами (только затронутые событиями регионы)
- Текущая дипломатическая ситуация (союзы, войны, отношения)
- Текущая военная ситуация (активные фронты, армии)
- Действия игрока за предыдущий ход
- Предыдущие события (последние 3-5)
- Доступные технологии для текущего года

**Опциональные данные (включать при необходимости):**
- Полная экономика только для крупных стран (топ 10 по ВВП)
- Детальная военная ситуация только для активных фронтов
- Ресурсы регионов только при строительстве
- Население регионов только при демографических событиях

**Принципы формирования контекста:**
- Не передавать полный мир в контекст
- Передавать только релевантные данные для текущего хода
- Приоритет данным, связанным с действиями игрока
- Сжимать данные, используя агрегацию вместо детализации
